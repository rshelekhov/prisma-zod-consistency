**Split `note: skipped unregistered or unsatisfied rule(s)` into three actionable messages.**

The old generic note conflated two very different cases — a typo in `--rules` (e.g. `R99`) and the normal case of live-DB rules being skipped because `--db` wasn't passed. As a result, a typo in CI configuration passed silently with exit=0.

Now:

- **Unknown rule id** (typo or rule from a future version) → `prisma-zod-consistency: unknown rule(s): R99` on stderr, with the list of registered rules, and **exit code 2** (bad invocation). Closes the silent-pass loophole.
- **Live-DB rule without `--db`** (R07-R09d in the default static run) → `info: live-DB rules skipped (pass --db to enable): R07, R08, R09, R09b, R09c, R09d`. Same exit code as before (no change).
- **Rule disabled via `.prismazodrc.json`** (`severity: "off"`) → silent. The user turned it off themselves; no note needed.

**Internal API change (pre-1.0, no external consumers known):** `RunResult.skippedRules` shape went from `RuleId[]` to `{ id: string; reason: "unknown" | "needs-db" | "disabled" }[]`. Only `packages/cli/src/index.ts` reads this field; JSON / SARIF output is unaffected.
